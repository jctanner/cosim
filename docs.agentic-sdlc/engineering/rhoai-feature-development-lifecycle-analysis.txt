# RHOAI Feature Development Lifecycle Analysis
**Complete Investigation: Official Process vs. Actual Practice**

**Date:** March 11, 2026
**Investigation Scope:** RFE → GA lifecycle for Red Hat OpenShift AI (RHOAI)
**Key Finding:** Architecture documentation lags product release by 3+ months, features shipping without required architectural artifacts

---

## Executive Summary

This investigation reveals **significant gaps between the official RHOAI feature development process and actual practice**, particularly around architecture documentation. While product documentation (v3.3 GA, v3.4 EA1 shipped) remains current, architecture documentation (v2.13, last updated December 2025) lags **3+ months behind** product releases.

**Update**: A **DRAFT** process improvement framework called "From Chaos to Clarity" was developed for RHOAI 3.4 EA1 MVP (target approval: January 23, 2026). This proposed framework addresses several identified gaps by introducing FDoR (Feature Definition of Ready) gates requiring architectural sign-offs and mandatory ADRs for features lacking clear implementation paths. However, the draft proposal does not fully resolve the disconnect between RFE Council's `requires_architecture_review` label and the enforcement mechanism, nor does it address the broken RHAIRFE→RHAISTRAT automation.

**Update (March 24, 2026)**: As of March 24, 2026, there is no evidence that "From Chaos to Clarity" was ever formally approved. The January 23, 2026 approval deadline appears to have passed without action. No Slack messages reference the framework by name. This means the FPDoR gates, RPDoD process, and 40/40/20 capacity allocation documented as "DRAFT" below likely remain DRAFT — all "partially addressed" gaps in this analysis are effectively still unaddressed in practice.

**Update (March 18, 2026)**: Catherine Weeks (Red Hat AI Leadership) sent a governance clarification to ai-eng-all@redhat.com ([Ownership and Commitment Guidelines for Red Hat AI](https://docs.google.com/document/d/1hz1yTZEPKuAkha53Aj9OeyXWx6VC0OBKhZ_sc7W1YWw/edit)) directly addressing the pressure pattern and ownership ambiguity documented in this analysis. Establishes: BU owns WHAT/WHY, Engineering owns HOW/WHEN. RFE Council approval = "green light to refine," not delivery mandate. Teams empowered to say "no" or "not this scope." Features target < 4 sprints. Includes a formal Decision Rights Matrix (RACI-like) covering feasibility, scope, priority, timeline, capacity, implementation approach, and acceptance criteria. See `google_docs/Clarifying Scoping and Eng Commitment Decisions for RHAI Features.eml`.

**Agentic SDLC context (March 18, 2026)**: On the same day, CTO Chris Wright and SVP Ashesh Badani mandated an Agentic SDLC for all of Global Engineering. The AI first pilot aims to use agents throughout the delivery pipeline documented here (RF creation → JER → refinement → development → docs → quality). Catherine's Decision Rights Matrix defines the human governance gates that agents must integrate with — an agent can help create an RF, but that doesn't commit engineering to deliver it. The many human decision points in this lifecycle (particularly feasibility assessment, scope negotiation, capacity allocation) are the steps least amenable to pure automation and most important to preserve. See `notes/RHOAI-ARCHITECTURE-DIAGRAMS.md` for full timeline.

**Agentic SDLC in practice (March 24, 2026)**: Jessica Forrester (RFE Council Co-Chair) is actively building an AI-powered "RFE builder" agent (#wg-rhai-rfe-builder) that "already uses the architecture context and Jason's RFE assesser as two independent adversarial reviewers." This is the first concrete evidence of agents being integrated into the lifecycle documented here, specifically at the RFE creation stage (Stage 1). Meanwhile, engineer feedback from the Global Engineering All-Hands (March 24) highlights an "Agentic Paradox" — AI has doubled raw code output but created 154% larger PRs with a "Review Tax" slowing senior engineers by ~20%, raising concerns about quality control bottlenecks at the human governance gates this analysis identifies as critical.

### Critical Findings

1. **Broken RFE Automation**: RHAIRFE→RHAISTRAT auto-cloning disabled (Jan 2025) due to duplicate creation issues, requiring manual PM intervention — now **15 months broken** with no fix in progress - ⚠️ **Not addressed in draft proposal**
2. **Architecture Review Label Exists But Not Enforced**: RFE Council sets `requires_architecture_review` label during assessment, but no tracking or gates ensure the review actually happens - 🔄 **Partially addressed by draft FDoR gate, but connection not defined**
3. **Architecture Documentation Debt**: 28+ "Feature documented in architecture diagrams" tickets exist as retroactive cleanup work
4. **Process Inversion**: Product docs are written BEFORE architecture docs, reversing the intended workflow - 🔄 **Draft FDoR gate aims to fix this**
5. **Missing Gates**: No enforced gates prevent features from shipping to GA without architecture documentation - 🔄 **Draft FDoR gate (10 weeks) and RPDoD gate (8 weeks) aim to fix this**
6. **ADR Optionality**: Architecture Decision Records are marked "optional" despite being critical for architectural decisions - 🔄 **Draft epic 0 process makes ADRs mandatory for unclear features**
7. **Automation Risk**: RHOAIENG-52636 (AI automation for architecture) could codify broken processes if not carefully designed - 🔄 **Draft includes AI automation for JIRA hygiene, not architecture generation**

**Legend**: ⚠️ Not addressed | 🔄 Partially addressed in DRAFT proposal | ✅ Fully addressed

## Lifecycle Stages: Official Process

### Phase 1: Request and Approval (Pre-Development)

#### 1.1 RFE (Request for Enhancement)
**Owner:** Business Unit / Product Management / Community
**Jira Project:** RHAIRFE (Red Hat AI RFE)
**Workflow:** New → Stakeholder review → Review → Pending approval → Approved

**Who Creates the RFE:**
- **Reporter** can be:
  - Product Manager (e.g., Rob Greenberg)
  - Field/Customer-facing teams (e.g., Brian Ball)
  - Engineering (e.g., Adel Zaalouk)
  - Community members

**Business Justification - Where It Goes:**

The business justification is written **in the Description field** of the RHAIRFE Jira ticket, typically structured as:

**Common Description Sections:**
- **Problem Statement** - What issue needs solving
- **Overview/Goals** - What we want to achieve
- **Proposed Solution/Rationale** - How to solve it
- **Value Proposition** - Business value (customer impact, revenue, competitive position)
- **Acceptance Criteria** - Definition of done

**Examples from actual RFEs:**

*RHAIRFE-1402 (Docling OCR):*
- In Description: "Problem Statement", "Proposed Solution/Rationale"
- In Comments: "This is an urgent blocker for Neuberger-Berman", "Q2 revenue potential", "required for GM Financial"

*RHAIRFE-1392 (vLLM config):*
- In Description: Explicit "Business justification: Red Hat is behind competitors with default config but matches or exceeds performance with optimal config. Prospects like Airbnb needs vllm optimal config."

*RHAIRFE-1350 (Kagenti cleanup):*
- In Description: "These unnecessary dependencies increase the stack size, complicate deployment, and will create significant tech debt if not addressed before midstreaming"

**Format:**
- ❌ NOT a separate Jira field called "Business Justification"
- ✅ Free-form text embedded in the Description narrative
- ✅ Sometimes explicit section titled "Business justification:" or "Value Proposition:"
- ✅ Often woven into Problem/Solution/Goals sections
- ✅ Supplemented by stakeholder comments adding customer names, revenue impact, urgency

**Requirements:**
- Business justification and alignment with Red Hat AI Outcomes (in Description)
- Priority field set by PM (Blocker, Critical, Major, Normal, Minor)
- Customer/market validation (in Description and Comments)
- RFE Council assessment and approval

**Note on RICE Scoring**:
- ❌ **NOT done at RFE stage** (RHAIRFE)
- ✅ **Done AFTER cloning to RHAISTRAT Feature** (part of FPDoR R3 requirement)
- RFE Council uses **Priority field** for ordering review queue, not RICE scores

**RFE Council Assessment Process:**

**Council Structure:**
- Co-Chairs: Jessica Forrester (Chief Architect), Jason Greene (Chief Architect)
- 40+ members: Architects, Staff Engineers, Tech Leads across Red Hat AI portfolio
- Time allocation: 1-2 hours per week per member
- Communication: #wg-redhat-ai-rfe-review Slack channel

**Assessment Criteria:**
1. RFE contains sufficient information for assessment
2. RFE is focused on one self-contained value feature (not too broad)
3. Description is objective and concise
4. Aligns with product roadmap, company strategy, product scope
5. Technically feasible
6. Potential impact on existing features/systems
7. Customer demand and business value

**RFE Council Does NOT Use RICE**:
- RFE Council reviews RFEs in **RHAIRFE** project
- Uses **Priority field** (Blocker, Critical, Major, Normal, Minor) set by PM for ordering
- Provides qualitative assessment (see template above)
- Does NOT calculate RICE scores

**RICE Scoring Happens Later** (on RHAISTRAT Feature):

After RFE approval and cloning to RHAISTRAT, the Feature undergoes RICE scoring as part of FPDoR R3.

**Formula**: `RICE Score = (Reach × Impact × Confidence) ÷ Effort`

**Who Scores**:
- **Reach, Impact, Confidence**: PMs and UX confer
- **Effort**: Engineering delivery leads and partners with variable impact

**When**: Before Planning Freeze (10 weeks before release) - FPDoR R3 requirement

**RHOAI-Specific Scoring Rubric**:

**Reach** (market impact - % of users affected in agreed time period):
- 1 = Very few users (niche request)
- 3 = Small user group (<10% of users)
- 5 = Moderate user group (10-30% of users)
- 8 = Large group (30-70% of users)
- 13 = Nearly all users (>70%), game changer for industry

**Impact** (customer/system benefit):
- 1 = Minimal (small QoL improvement)
- 3 = Low (slight improvement in workflow or minor feature)
- 5 = Medium (noticeable productivity or revenue gain)
- 8 = High (major pain point solved, new capability)
- 13 = Massive (game-changer, strategic differentiator) or Non-negotiable (legal requirement)

**Confidence** (certainty of Reach/Impact estimates):
- 50% = Mostly hypothesis or guesswork
- 75% = Some data, but still assumptions
- 100% = Strong evidence/data (validated by analytics or customer feedback)

**Effort** (high-level, relative, NOT person-months):
- 1 = Low effort (few unknowns, one team, one sprint)
- 2 = Medium effort (some unknowns, couple teams, maybe >1 sprint)
- 3 = Medium+ (some unknowns, cross-team, requires feature mapping once prioritized)
- 5 = High effort (multiple unknowns, multiple teams, will require feature mapping)
- 8 = Massive effort (multiple unknowns/dependencies, work outside org, Story mapping required)
- 13 = Too large to responsibly estimate (MUST be broken into smaller features)

**Jira Integration**:
- RHAISTRAT and RHOAISTRAT projects have RICE fields built-in
- "Prioritization" tab appears when any field has entry
- RICE score auto-calculated when all four fields populated
- Prioritization number must be visible in Jira issue (per FPDoR R3)

**Best Practice**: Add comment starting with "RICE SCORE JUSTIFICATION:" to document scoring rationale

**Connection to FPDoR**:
- FPDoR R3 requires "RICE score is completed and connected to feature sizing"
- Feature sizing (T-shirt: S/M/L/XL) must align with Effort score
- L (9 weeks) or XL (12 weeks) features must be broken down into smaller increments before FPDoR submission

**Official Assessment Template (ACCEPTED):**
Council members add PRIVATE (Red Hat Employee only) comment with:
```
Assessment: Accepted
Requires architecture review: [Yes/No]
Primary component: <the component team that will lead implementation>
Impacted components: <other component teams that will contribute>
High level architecture plan: <architectural overview for implementation>
Other related work: <list related features, tasks, dependencies>
```

**Critical Field**: **"Requires architecture review: [Yes/No]"**
- If "Yes", council sets label `requires_architecture_review`
- ✅ **CONNECTION FOUND**: FPDoR R4 (see Proposed Process Improvements section) requires Architecture Forum approval documentation for features requiring review
- ⚠️ **PARTIAL GAP**: Both RFE Council Charter and FPDoR are documents in draft/proposal state

**Official Assessment Template (DECLINED):**
Council members add PRIVATE comment with:
```
Assessment: Declined
Justification: <detailed justification. If remedy is scope adjustments, identify next steps>
```

**Official Jira Workflow:**
```
New → Stakeholder Review → Review → Pending Approval → Approved
                                  ↘ Rejection Pending → Closed
```

- **New**: Default when PM creates RFE
- **Stakeholder Review**: PM submits to council for review
- **Review**: Council member reviewing (assigns to self)
- **Pending Approval**: Council accepted RFE
- **Approved**: PM approved, RFE moves into development
- **Rejection Pending**: Council rejected RFE
- **Closed**: PM has closed RFE

**Examples:**
- RHAIRFE-1402: "AutoML for tabular data" (Approved)
- RHAIRFE-1392: "Support for LLaMA Stack" (Approved)

**Clone to RHAISTRAT (Feature or Initiative):**

**Official Process (RFE Council Charter):**
> "If the RFE has been approved, clone or create linked Feature tickets and assign them to the primary owner team identified by the RFE Council."

**Target Jira Projects:**
- **RHAISTRAT**: All work items across Red Hat AI (rhoai, rhelai, rhaiis) - **PRIMARY**
- **AIPCC**: AIPCC-specific work items only

**RHAISTRAT Issue Types** (2 types):

| Issue Type | Purpose | Release Type Field | Examples |
|------------|---------|-------------------|----------|
| **Feature** | Customer-facing capability delivering business value | Dev Preview, Tech Preview, or GA | New user-facing functionality, API enhancements |
| **Initiative** | Technical/product improvements (stability, tooling, tech debt) | N/A (not applicable) | Code cleanup, dependency updates, infrastructure improvements |

**Which Type to Use:**
- **Customer-facing RFE** → Clone as **Feature**
- **Technical/internal RFE** → Clone as **Initiative**
- Example: RHAIRFE-1350 "Kagenti Codebase Cleanup" → Would become **Initiative** (not customer-facing)
- Example: RHAIRFE-1402 "Docling OCR" → Would become **Feature** (customer-facing)

**Official Guidance:**
- PM clones approved RFE into corresponding RHAISTRAT issue type
- Assigned to primary component team identified by RFE Council
- For large RFEs: Recommended to break into multiple Features/Initiatives each with standalone value

**Both Issue Types**:
- ✅ Follow same FPDoR checklist (all 9 requirements including RICE scoring)
- ✅ Subject to same JIRA hygiene standards
- ✅ Go through same Planning Freeze → RPDoD process
- ✅ Weekly review required if status is "in progress" or "refinement"

**Historical Note on Automation:**
- **December 2024:** Automation implemented to auto-clone approved RFEs from RHAIRFE to RHAISTRAT
- **January 2025:** Automation **DISABLED** due to creating duplicate tickets
- **Current Process:** PMs must **manually clone** approved RFEs to RHAISTRAT
- ⚠️ **GAP**: Official charter doesn't document the automation failure or current manual process

**Evidence:**
> "The RHAIRFE-> RHAISTRAT automation is disabled. Communications was sent this morning stating the same." (Slack, January 2025)

#### 1.2 Feature or Initiative (RHAISTRAT)
**Owner:** Product Management
**Jira Project:** RHAISTRAT
**Issue Types:** Feature (customer-facing) OR Initiative (technical/product improvements)
**Source:** Cloned from approved RHAIRFE ticket

**Common Requirements (Both Types):**
- Must be **approved by Architect Council** before proceeding (NON-NEGOTIABLE per RHOAIENG-52657)
- Links to associated RFE (parent RHAIRFE ticket)
- Target release version (e.g., rhoai-3.4.EA2)
- Must meet FPDoR (Feature Planning Definition of Ready) with all 9 requirements before Planning Freeze

**Feature-Specific Requirements:**
- Support level designation: Dev Preview (DP), Tech Preview (TP), or GA
- Release Type field populated

**Initiative-Specific:**
- Release Type field: N/A (not applicable)
- Focused on technical value (stability, tooling, tech debt) rather than customer-facing features

**Example Feature:** RHAISTRAT-1081 "MLflow integration in RHOAI TP"
- Type: Feature (customer-facing)
- Created: 2026-01-07
- Status: Review (updated March 24, 2026; previously In Progress)
- Target: rhoai-3.4.EA2
- Release Type: Tech Preview
- Has refinement document link

**Example Initiative:** RHAISTRAT-XXXX "Kagenti Codebase Cleanup" (hypothetical)
- Type: Initiative (technical debt)
- Release Type: N/A
- Focused on reducing dependencies and technical debt

## Gap Analysis with Evidence

### Gap 1: Architecture Documentation Treated as Optional

**Official Process:**
> "(optional) When warranted, complete an Architecture Decision Record"
> "(Recommended) Begin/Complete the Architecture Design Document"

**Problem:** The words "optional" and "recommended" mean these artifacts are frequently skipped

**Evidence:**
- 28 retroactive "Feature documented in architecture diagrams" tickets
- AutoML ADR exists but not integrated 2 months later
- LlamaStack has no ADR despite being platform-critical
- arch-overview.md is 3 versions behind product release

**Impact:**
- New engineers cannot understand architectural decisions
- Technical debt accumulates invisibly
- Cross-component integration suffers from lack of shared understanding

### Gap 2: No Enforcement Mechanism for Architecture Artifacts

**RFE Council Sets Architecture Review Flag, But No Follow-Through:**

**What EXISTS (from RFE Council Charter):**
- ✅ RFE Council assessment includes: "Requires architecture review: [Yes/No]"
- ✅ If "Yes", council sets label `requires_architecture_review` on RFE
- ✅ Label indicates "design of this feature should be reviewed in the Architecture Forum"

**What's MISSING:**
- ❌ No tracking of whether architecture review actually happens
- ❌ No gate preventing feature from proceeding if review required but not completed
- ❌ No process documented for conducting the architecture review
- ❌ No field to mark "Architecture Review Complete"
- ❌ No automated check that ADRs exist for features with `requires_architecture_review` label

**Missing Gates:**
- ❌ No gate between "Development" and "Dev Preview" requiring architecture review completion
- ❌ No gate between "Tech Preview" and "GA" requiring arch-overview.md update
- ❌ No Jira workflow transition blocked if `requires_architecture_review` label present but review not done

**Evidence:**
From template epic RHOAIENG-52657:
- **Required:** "An approved RFE by the Architect Council"
- **Missing:** Any requirement for architecture documentation OR completion of architecture review

**Comparison to Other Gates:**
- ✅ Security review IS required (ProdSec sign-off)
- ✅ Definition of Ready IS enforced
- ✅ RFE Council approval IS required
- ⚠️ Architecture review flag IS set by RFE Council
- ❌ Architecture review completion is NOT tracked or enforced
- ❌ Architecture documentation is NOT required

### Gap 3: Ownership Ambiguity for Architecture Artifacts

**Who creates architecture diagrams?**
- Official process: "Begin/Complete the Architecture Diagram" (no owner specified)
- ADR README: Stored in arch-decision-records repository (suggests Architecture team)
- Refinement guidelines: No mention of who updates arch-overview.md

**Evidence of Confusion:**
- RHOAIENG-52636 created to "automate" architecture generation
- 28 CLONE tickets suggest centralized cleanup effort (not component teams doing it)
- Slack discussions about "architecture review" but unclear process

**Result:** Everyone assumes someone else will do it

**Update (March 18, 2026)**: Catherine Weeks' governance clarification addresses ownership ambiguity at the feature-level decision-making layer, though not specifically for architecture artifacts:

| Decision | Owner |
|----------|-------|
| RFE technically feasible | RFE Council (Decide) |
| RFE too broad, needs split | RFE Council (Decide), Engineering Team (Recommend) |
| Feature business priority | PM (Decide) |
| Final feature scope | Delivery Owner (Decide), PM/Tech Owner/Eng Team (Approve) |
| Implementation timeline | Delivery Owner (Decide), Eng Team (Approve) |
| Team capacity allocation | Engineering Team (Decide) |
| Implementation approach | Technical Owner (Decide), Delivery Owner (Approve) |
| Acceptance criteria met | PM (Decide) |

This clarifies who decides WHAT gets built and WHEN, but the gap around who creates/maintains architecture *artifacts* (ADRs, diagrams, arch-overview.md) remains unaddressed. The architecture artifact ownership problem exists at a layer below the feature decision rights — it's about the work products required during implementation, not the decisions about scope and priority.

### Gap 4: Process Timing Encourages Shortcuts

**Official Timeline:**
> "NOTE: START THIS STEP AT LEAST TWO WEEKS IN ADVANCE OF PLANNED MERGE INTO RHOAI"

**Problem:** Architecture work happens AFTER implementation decisions are made

**Evidence:**
- Product docs (v3.3) are current
- Architecture docs (v2.13) are 3 months behind
- This is mathematically impossible if architecture docs come FIRST

**Actual Workflow:**
1. Feature approved
2. Implementation starts
3. Product docs written (for release)
4. (maybe someday) Architecture docs updated

### Gap 5: Broken RFE-to-Feature Automation

**RHAIRFE → RHAISTRAT Cloning Process**

**Official Process (Intended):**
- RFE approved in RHAIRFE
- Automation automatically clones to RHAISTRAT as Feature ticket
- PM continues with Feature workflow

**What Actually Happened:**
1. **December 2024:** Automation implemented to auto-clone approved RFEs from RHAIRFE to RHAISTRAT
2. **January 2025:** Automation **DISABLED** due to creating duplicate tickets
3. **Current Process:** PMs must **manually clone** each approved RFE to RHAISTRAT

**Evidence:**
> "The RHAIRFE-> RHAISTRAT automation is disabled. Communications was sent this morning stating the same." (Slack, January 2025)

**Impact:**
- ❌ Manual process prone to errors and delays
- ❌ No guaranteed linkage between RHAIRFE and RHAISTRAT tickets
- ❌ PMs must remember to clone after approval
- ❌ Potential for approved RFEs to "fall through the cracks"
- ❌ Defeats the purpose of automation initiative

**Root Cause:** Duplicate detection logic failed, automation disabled rather than fixed

**Recommendation:** Fix automation duplicate detection and re-enable with proper safeguards

### Gap 6: Automation Before Process Fix

**RHOAIENG-52636: AI Automation for RHOAI architecture**

  - Strategic alignment evaluation

## Post-Release: Support Lifecycle Page

Once a feature ships in an RHOAI release, that release enters the support lifecycle defined on the customer-facing [RHOAI Self-Managed Life Cycle page](https://access.redhat.com/support/policy/updates/rhoai-sm/lifecycle). This is the end-of-pipe output of the feature development process documented above — the lifecycle page answers "how long do we support this release?" but does not track individual feature support levels within a release.

### Release Categories

RHOAI Self-Managed follows three release category models:

| Category | Full Support Duration | Extended Support | Cadence |
|----------|----------------------|------------------|---------|
| **Fast** | 1 month (or until next fast release) | None | Every minor release |
| **Stable** | 7 months | None | Every 3 minor releases |
| **EUS** (Extended Update Support) | 7 months | 11 months additional | Every 9 minor releases |

During Full Support, Red Hat releases qualified Critical and Important Security Advisories. During EUS phases, Red Hat maintains component-specific support.

**Recent change (RHOAI 3.x)**: The versioning model changed with RHOAI 3.4. What were previously "fast releases" (3.0, 3.1, 3.2) are now called **"Early Access" (EA) drops** (3.4.0-ea.1, 3.4.0-ea.2) shipped to a beta channel, with the stable version (3.4.0) going to the stable channel. Per Rishab Prasad (#forum-ocp-operator-fw, Feb 2026): "Red Hat AI recently moved to a new Early Access (EA) versioning model where we would be doing EA drops to the beta channel and the stable version to the stable channel."

**EA releases are NOT supported releases**: Per Andrew Ballantyne (#feature-openshift-ai-nim): "Early Access releases are 'check behind the curtain' releases, not supported releases. Thus no 'upgrade path' exists for them. No 'EA1 or EA2' upgrading to 'GA' stable releases — ever."

### Who Maintains the Lifecycle Page

**Primary owner: Ignacio Lago** (Release Engineering / PLM)

**Evidence:**
- Posts drafts for review in release working group Slack channels before publishing
  - Example (#wg-3_3-openshift-ai-release): "Draft for the Life Cycle pages updated for 3.3... If everyone is fine, it will be published later today."
- Uses a staging URL for drafts: `dxp-cphb-prod.apps.ext-waf.drop.prod.us-west-2.aws.paas.redhat.com/support/policy/updates/rhoai-sm/lifecycle`
- Tagged directly by others when updates are needed (Rishab Prasad, jalberts, mansaldo)
- Collects component support data from individual teams via Google Doc templates

**Tracking:** PLMCORE JIRA tickets are cloned per release to track lifecycle page updates:
- PLMCORE-13492: Lifecycle update for RHOAI 2.24
- PLMCORE-13489: Lifecycle update for RHOAI 2.23
- PLMCORE-8979: Original RHOAI Lifecycle Proposal (March 2024, reviewed by Jason Sherman / PLM Product Operations)

**Ownership gap:** Despite Ignacio being the de facto owner, this is not widely known:
- Trevor Royer (#forum-openshift-ai): "Looks like it is just out of date. I'm not sure who owns that page though...."
- Wenliang (#forum-ai-core-platform): "Does anyone know who is maintaining this page?"

### How the Data Is Decided

**1. Release type designation (Fast/Stable/EUS):**
- Decided by PM/BU leadership, not engineering
- The lifecycle policy structure was initially proposed via PLMCORE-8979 (March 2024)
- Cadence rules (stable every 3, EUS every 9) are embedded in the policy itself
- Individual release type assignment follows these rules mechanically

**2. GA dates and support windows:**
- GA dates come from the release planning process (tracked in Smartsheet)
- Support end dates are calculated from the release category rules (e.g., stable = GA + 7 months)
- Ignacio updates the page once a release ships

**3. Supported OCP versions:**
- Requires engineering input — manually assembled and error-prone
- Example: Jeff DeMoss caught that the 3.3 lifecycle page listed OCP 4.20 & 4.21 but was missing 4.19, which the release notes listed as supported

**4. Component-level support:**
- Tracked on a separate page: [RHOAI Supported Configurations](https://access.redhat.com/articles/rhoai-supported-configs)
- Also maintained by Ignacio Lago, who collects data from teams via Google Doc templates
- Component support can differ from release support — e.g., Data Science Pipelines was supported in RHOAI 2.8 GA but deprecated in RHOAI 2.8 EUS
- Current discussion: whether llama-stack (TP in 2.25) should be supported in 2.25 EUS given that fixes primarily target latest versions

**5. Upgrade paths:**
- The page defines which upgrade paths are tested and supported
- OLM (Operator Lifecycle Manager) rules enforce which upgrades are technically possible
- Direct upgrades from RHOAI 2.25 or earlier to 3.3 are NOT currently supported due to "significant architectural changes"
- Significant confusion exists around which paths are actually tested vs technically possible

### Known Problems

**1. Page frequently out of date:**
- Jan Stourac encountered a subscription-required access error
- Multiple people have noted content being stale
- The "supported graphic" updates automatically but the text content is manually maintained

**2. EUS confusion with new versioning model:**
- Existing rule: "EUS release every nine minor releases"
- With EA versioning, this rule may not apply the same way
- Gryan (#wg-2-25-z-3-3-1-rhoai-upgrade-project): "Are we still doing EUS releases in 3.x? There's nothing in the smartsheet to show which is the next one. If 3.9 is not the next EUS, then the following in the lifecycle doc probably needs to be changed."

**3. Entirely manual process:**
- No automation for collecting component data, OCP compatibility info, or upgrade path validation
- Single person (Ignacio) responsible for updates across all releases
- Data collection requires manually pinging individual component teams

**4. Defines cadence, not content:**
- The page tells customers HOW LONG a release is supported
- Says nothing about WHAT'S IN a release
- As noted in earlier analysis: "this is release cadence, not content" — the lifecycle page is release infrastructure, not product strategy

**5. Upgrade path documentation vs reality gap:**
- The page states: "Red Hat tests and supports upgrade paths that are allowed according to the OLM rules"
- But also provides specific supported upgrade path examples that may not cover all OLM-allowed paths
- Engineers have raised concerns about whether all documented paths are actually tested

### Connection to Feature Lifecycle

The lifecycle page represents the **final stage** of the feature development lifecycle documented in this analysis:

```
RFE → RHAISTRAT → Refinement → Implementation → Release → LIFECYCLE PAGE
                                                              │
                                                              ├── How long is this release supported?
                                                              ├── What OCP versions work with it?
                                                              ├── What upgrade paths are supported?
                                                              └── What components are supported?
                                                                    (separate Supported Configs page)
```

**Key observation:** The same pattern of manual processes, unclear ownership, and documentation lag seen in the architecture documentation gap (arch-overview.md lagging 3+ months) also appears in the lifecycle page maintenance. Both are customer-facing artifacts that depend on a single person manually collecting data from distributed teams with no enforcement mechanism to ensure accuracy or timeliness.

### 2. **Fix RHAIRFE→RHAISTRAT Automation**

**Status**: ⚠️ **NOT addressed in DRAFT**

**Current Problem:** Automation to clone approved RFEs from RHAIRFE to RHAISTRAT was disabled in January 2025 due to creating duplicates. PMs now manually clone approved RFEs, creating risk of:
- Approved RFEs falling through the cracks
- No guaranteed linkage between RHAIRFE and RHAISTRAT tickets
- Manual process prone to delays and errors
- Inconsistent tracking of feature lineage

**Root Cause:** Duplicate detection logic failed when automation attempted to clone already-cloned RFEs

**Recommended Fix:**

**Phase 1: Immediate (Week 1-2)**
- Review existing automation code to understand duplicate creation issue
- Implement proper duplicate detection:
  - Check for existing RHAISTRAT tickets with same RHAIRFE parent
  - Add "RHAIRFE Parent" field to RHAISTRAT for bidirectional linking
  - Only clone if no existing RHAISTRAT ticket with same parent link

**Phase 2: Enhanced Automation (Month 1)**
- Re-enable automation with safeguards:
  - Dry-run mode for 2 weeks to validate no duplicates
  - Add notification to PM when auto-clone succeeds
  - Add alert to PM if auto-clone skipped due to existing ticket
  - Log all clone attempts for audit trail

**Phase 3: Validation (Month 2)**
- Monitor for 1 month to ensure no duplicates
- Validate 100% of approved RFEs are being cloned
- Measure time from "Approved" status to RHAISTRAT creation
- Get PM feedback on automation effectiveness

**Success Metrics:**
- ✅ 0 duplicate RHAISTRAT tickets created
- ✅ 100% of approved RHAIRFE tickets auto-cloned within 1 hour
- ✅ Bidirectional linkage between RHAIRFE ↔ RHAISTRAT maintained
- ✅ PMs report reduced manual work

**Fallback Plan:** If automation cannot be fixed reliably within 1 month, implement manual checklist in RHAIRFE approval workflow that blocks "Approved" status until PM confirms RHAISTRAT cloning complete

### 3. **Enforce Architecture-Before-Implementation Workflow**

**Status**: ✅ **Substantially addressed in DRAFT**

- ✅ DRAFT RPDoD gate at 8 weeks requires stakeholder sign-offs including architectural approval
- ❌ DRAFT doesn't specify arch-overview.md update deadline
- ❌ DRAFT doesn't define connection to `requires_architecture_review` label

|-------|------|-------------------|--------|

| RPDoD → Implementation | Stakeholder Sign-Off | PM/UX/Docs/Pillar approval | ✅ In DRAFT |
| Tech Preview → GA | Architecture Complete | arch-overview.md updated | ⚠️ ADD TO DRAFT |

**Specific Actions:**

**Leverage Existing `requires_architecture_review` Label:**
1. **Add Jira Field:** "Architecture Review Complete" (checkbox)
2. **Add Jira Field:** "Architecture Review Date" (date field)
3. **Add Jira Field:** "Architecture Forum Link" (URL to Architecture Forum discussion/approval)
4. **Create Jira Automation Rule:**
   - IF label `requires_architecture_review` is present
   - AND "Architecture Review Complete" is unchecked
   - THEN block transition to "In Progress" or "Dev Preview"
   - WITH error message: "Architecture review required but not completed. See Architecture Forum guidelines."

**Workflow Gates:**

  - If RFE has `requires_architecture_review` label → Architecture review MUST be completed
  - If feature introduces new component or changes architecture → ADR MUST be created and approved
- **Before Tech Preview:** Architecture diagrams MUST be created (in ADR repo)

### 4. **Clarify Ownership and Accountability**

**Status**: 🔄 **Partially addressed by Catherine Weeks' governance clarification (March 18, 2026)**
- ✅ Feature-level decision rights now formally defined (BU owns WHAT/WHY, Engineering owns HOW/WHEN)
- ✅ Teams empowered to negotiate scope and say "no"
- ✅ Delivery Owner role clarified as scope/timeline decision maker
- ✅ Technical Owner role clarified as implementation approach decision maker
- ❌ Architecture *artifact* ownership still undefined (who creates ADRs, diagrams, arch-overview.md)
- ❌ No mapping between Decision Rights Matrix roles and architecture documentation responsibilities

**Current Problem:** "Everyone's responsibility = No one's responsibility"

**Proposed Ownership Model:**

| Artifact | Creator | Reviewer | Maintainer |
|----------|---------|----------|------------|
| ADR | Technical Lead (feature) | SPSE Architect | Architecture Council |
| Architecture Diagrams | Component Team | SPSE Architect | Architecture Team |
| arch-overview.md | Architecture Team | Architecture Council | Architecture Team |
| Product Docs | Documentation Team | PM + Architects | Documentation Team |

**Accountability Mechanism:**
- Feature Owner: Ensure ADR created and approved before TP
- Technical Lead: Create initial architecture diagrams
- Architecture Team: Integrate diagrams into arch-overview.md within 1 sprint of TP
- SPSE Architect: Sign-off required for both ADR and diagrams

## Conclusion

### Summary of Gaps

| Issue | Official Process | Actual Practice | Impact |
|-------|-----------------|-----------------|--------|
| **RFE Automation** | Auto-clone approved RFEs to RHAISTRAT | Manual cloning (automation disabled Jan 2025 — 15 months broken) | Approved RFEs can fall through cracks, inconsistent tracking |
| **Architecture Review Flag** | RFE Council sets `requires_architecture_review` label | Label set but no tracking/enforcement of review completion | Features flagged for review can proceed without it |
| **Architecture Documentation Timing** | Should be created during Design phase | Created retroactively after implementation | ADRs don't inform decisions, just document them after |
| **Architecture Diagram Currency** | Should be current with product | 3+ months behind (v2.13 vs v3.3 GA / v3.4 EA1) | Engineers lack current reference architecture |
| **ADR Requirements** | "Optional" | Skipped frequently | Architectural decisions undocumented |
| **Process Enforcement** | No gates for architecture docs | Can ship to GA without arch docs | Technical debt accumulates |
| **Ownership** | Unclear who updates arch-overview.md | Everyone assumes others will do it | Work falls through cracks |
| **Feature Scoping Pressure** | RFE Council approval = refine, not commit | Teams felt mandated to deliver as approved | Over-commitment, burnout, scope creep — 🔄 **Addressed by Catherine Weeks' governance clarification (March 18, 2026)** |
| **Automation Focus** | Should enforce workflow | Risk of automating broken process | Could codify problems |

### Critical Success Factors for Fix

1. **Leadership Buy-In:** Architecture documentation must be treated as non-negotiable as security reviews
2. **Process Change:** Add gates that block progression without architecture artifacts
3. **Clear Ownership:** Assign specific roles for creation, review, and maintenance — Catherine Weeks' Decision Rights Matrix (March 18, 2026) establishes feature-level ownership (BU=WHAT/WHY, Engineering=HOW/WHEN) but architecture artifact ownership still needs equivalent clarity
4. **Tool Support:** Automation should VALIDATE and ASSIST, not REPLACE human judgment — reinforced by CTO Agentic SDLC mandate (March 18, 2026): "human judgment on architecture, design, and complex logic remains critical"
5. **Grandfather Clause:** Use existing CLONE tickets to catch up, but enforce going forward
6. **Agentic SDLC Integration:** As agents take on more of the delivery pipeline (RF creation → development → docs), the human governance gates documented here become the critical control points. Catherine's Decision Rights Matrix defines where agent-generated work requires human approval before proceeding.

### RHAISTRAT Issue Types - Quick Reference

**After RFE approval, PMs clone to RHAISTRAT as either:**

| Issue Type | Purpose | Release Type Field | When to Use | FPDoR? |
|------------|---------|-------------------|-------------|--------|

**Examples**:

- **Initiative**: "Kagenti Codebase Cleanup & Dependency Reduction" (from RHAIRFE-1350) - technical debt

### RICE Scoring - Quick Reference

**Formula**: `(Reach × Impact × Confidence) ÷ Effort`

| Factor | Scale | Description |
|--------|-------|-------------|
| **Reach** | 1, 3, 5, 8, 13 | % of users affected: 1=niche, 13=>70% (game changer) |
| **Impact** | 1, 3, 5, 8, 13 | Customer benefit: 1=minimal, 13=massive/non-negotiable |
| **Confidence** | 50%, 75%, 100% | Certainty of estimates: 50%=hypothesis, 100%=strong data |
| **Effort** | 1, 2, 3, 5, 8, 13 | Work required: 1=one team/sprint, 13=too large (break down) |

**Who Scores**: PMs + UX (Reach/Impact/Confidence), Engineering leads (Effort)
**Jira**: RHAISTRAT/RHOAISTRAT have built-in RICE fields, auto-calculation
**Best Practice**: Comment with "RICE SCORE JUSTIFICATION:" to document rationale
**FPDoR Connection**: R3 requires RICE score connected to feature sizing
